🔹 ТЕМА 1/4 · AI INFRASTRUCTURE
NVIDIA Vera Rubin: 256 CPU в одной liquid-cooled стойке

📌 ФАКТ
The Register сообщает о референс-дизайне NVIDIA с 256 CPU Vera в одной стойке с прямым жидкостным охлаждением (The Register, 16 марта 2026). Архитектура подразумевает интеграцию Vera CPU с Rubin GPU в составе rack-scale платформы следующего поколения после GB200/GB300 NVL72.

📍 КОНТЕКСТ
NVL72 текущего поколения объединяет 72 GPU и 36 Grace CPU в одной стойке (NVIDIA newsroom, 2024). Заявленная плотность CPU вырастает примерно в семь раз на стойку при сохранении формфактора rack-scale. Это смещает точку проектирования ЦОД: воздушное охлаждение и стойки 30–50 кВт выводятся из контура для топовых конфигураций, базовой становится плотность 120–180 кВт/стойка [оценка автора на основе публичных данных по GB200 NVL72 ~120 кВт].

🎯 РЕЛЕВАНТНОСТЬ ДЛЯ AI DISTRIBUTION PLATFORM
Высокая. Затрагивает BOM-генератор (новая платформенная SKU-линия с обязательным DLC-контуром), граф совместимости (исключение воздушных вариантов для Rubin-класса), TCO-калькулятор (CapEx на CDU, manifolds, dry-coolers становится первого порядка).

🔮 ПРОГНОЗ НА 12 МЕСЯЦЕВ
→ Низкий: задержка серийных поставок Vera/Rubin из-за yield HBM4 — рынок продолжает разворачивать GB300 NVL72, Vera-стойки появляются только в составе пилотов гиперскейлеров.
→ Базовый: ограниченная коммерческая доступность через Tier-1 ODM (Foxconn, Quanta, Wiwynn) с фокусом на neocloud-операторов; VAR-канал получает спецификации, но не storage конфигурации.
→ Высокий: Supermicro и Dell включают Vera-стойки в стандартный каталог к Q4 2026 с готовыми CDU-комплектами; B2G-тендеры начинают требовать соответствие плотности 150+ кВт/стойка как baseline.

❓ ОТКРЫТЫЕ ВОПРОСЫ
1. Полный TDP стойки и требования к температуре подачи теплоносителя — публичных данных нет.
2. Совместимость с существующими CDU от Vertiv, Motivair, CoolIT — требует подтверждения от вендоров.
3. Сценарий частичной заполненности стойки (degraded mode) — допустим ли с точки зрения гарантии NVIDIA.
4. Сетевая топология: NVLink-домен и выход на Ultra Ethernet / Quantum-X — детализация не опубликована.

📋 BACKLOG-ITEM
Название: Расширить граф совместимости категорией «rack-scale liquid-only»
Триггер: публикация Vera-стойки на 256 CPU (The Register, 16 марта 2026)
Затронутые компоненты: граф совместимости, BOM-генератор, TCO-калькулятор
Требования:
1. Ввести атрибут cooling_class={air, hybrid, DLC-mandatory} для каждой платформы — без этого BOM будет генерировать невалидные конфигурации для Rubin-класса.
2. Добавить в BOM обязательный раздел facility-side: CDU, manifold, QDC, дополнительный chiller — иначе VAR-партнёр получит заявку, не учитывающую 30–40% CapEx.
3. Добавить в TCO статью «вода и подготовка теплоносителя» (фильтрация, ингибиторы, контроль pH) — операционный пункт, который интеграторы регулярно упускают.
4. Параметризовать минимальную плотность стойки в кВт как фильтр совместимости с площадкой клиента — отсекает заведомо нереализуемые проекты на ранней стадии.

✍️ ЧЕРНОВИК ПОСТА
В высокоплотной инфраструктуре граница между «оборудованием» и «зданием» размывается. NVIDIA анонсирует стойку с 256 Vera CPU и Rubin GPU под прямое жидкостное охлаждение (The Register, март 2026). Это не инкремент к NVL72, а изменение точки проектирования: воздушный вариант для топ-класса больше не существует как опция.

Для дистрибутора это означает, что BOM перестаёт быть списком серверов. В спецификации появляется facility-side: CDU, manifolds, QDC, контур подготовки воды, dry-coolers. Без этих позиций конфигурация неработоспособна, а VAR-партнёр получает заявку с пропущенными 30–40% CapEx.

Отдельный пункт — сценарий частичного заполнения. Гиперскейлер развернёт стойку целиком; региональный интегратор — нет. Допустим ли degraded mode с точки зрения гарантии вендора, публично не описано.

Вопрос для обсуждения: на каком уровне абстракции граф совместимости должен учитывать параметры здания клиента — температуру подачи теплоносителя, доступную электрическую мощность ввода, высоту потолков машзала?

🔹 ТЕМА 2/4 · ACCELERATORS
Intel Crescent Island: отдельный GPU для inference

📌 ФАКТ
Intel представила data center GPU «Crescent Island», позиционируемый как inference-ориентированный ускоритель (HPCwire, 15 октября 2025). Параллельно SambaNova и Intel объявили партнёрство по AI inference (HPCwire, 9 апреля 2026).

📍 КОНТЕКСТ
После сворачивания Ponte Vecchio и неоднозначной траектории Gaudi 3 Intel переходит к продуктовой сегментации: training-класс остаётся под вопросом, inference получает отдельную линейку. Это совпадает с разделением рынка: NVIDIA сохраняет training-монополию, тогда как inference-сегмент фрагментируется (AMD MI355X, Groq, SambaNova, Cerebras, AWS Trainium/Inferentia).

🎯 РЕЛЕВАНТНОСТЬ ДЛЯ AI DISTRIBUTION PLATFORM
Средняя. Граф совместимости должен поддерживать inference-only ветвь (без NVLink-доменов, с другими требованиями к сети и памяти). BOM-генератор пока не имеет достаточных данных по Crescent Island для практической интеграции.

🔮 ПРОГНОЗ НА 12 МЕСЯЦЕВ
→ Низкий: Crescent Island задерживается до 2027 года, как Ponte Vecchio; SambaNova-партнёрство ограничивается совместными white papers.
→ Базовый: ограниченные поставки через OEM-партнёров (Dell, Supermicro) для специфических workloads — embedding-сервинг, классические рекомендательные модели, не LLM-первого ряда.
→ Высокий: государственный сектор США и ЕС включает Crescent Island в перечень акцептованных ускорителей по причинам suverenitet/EXPORT-control, формируя устойчивый B2G-канал независимо от технических характеристик.

❓ ОТКРЫТЫЕ ВОПРОСЫ
1. Реальная пропускная способность памяти и тип (HBM3e / LPDDR) — публичных спецификаций недостаточно.
2. Программная экосистема: OneAPI vs CUDA-совместимость через SYCL — насколько готовы фреймворки vLLM, TensorRT-LLM аналоги.
3. Цена за токен относительно H100/H200 — единственная метрика, по которой inference-ускоритель оценивается заказчиком.
4. Поддерживает ли архитектура disaggregated prefill/decode (как DeepSeek/Mooncake-сценарии).

✍️ ЧЕРНОВИК ПОСТА
Рынок ускорителей расщепляется. Training остаётся за NVIDIA, inference уходит в зону конкуренции, где Intel выводит отдельный продукт — Crescent Island (HPCwire, октябрь 2025), и одновременно объявляется альянс с SambaNova (HPCwire, апрель 2026).

Для архитектора это означает, что граф совместимости больше не может оперировать категорией «GPU». Inference-ускоритель и training-ускоритель — разные классы по сетевой топологии, требованиям к памяти, профилю питания и программному стеку. Конфигурация под embedding-сервис на Crescent Island и кластер для дообучения LLM на H200 имеют пересечение только по уровню стойки.

Отдельный сюжет — B2G. Государственные заказчики в США и ЕС всё чаще формулируют требования через критерий «не-NVIDIA» по соображениям суверенитета поставки. Это создаёт спрос, в котором технические характеристики вторичны.

Вопрос для дискуссии: должен ли граф совместимости различать ускорители по сценарию (training/inference/embedding) на верхнем уровне или это деление искусственно для следующего поколения?

Конкретные цифры по затронутым проектам в материале не приводятся [НЕ ВЕРИФИЦИРОВАНО — первичный материал не разобран].

📍 КОНТЕКСТ
В 2024–2025 несколько крупных проектов гиперскейлеров получили отказы или отсрочки на уровне муниципалитетов (Virginia, Georgia, Arizona — упомянуто в публичной прессе, конкретные кейсы требуют атрибуции).
